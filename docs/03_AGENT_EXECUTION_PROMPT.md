# 에이전트 실행 프롬프트 - Neon Pop Sweeper (v2.2)

아래 지시를 그대로 따르세요.

## 목표
HTML/CSS/Vanilla JavaScript와 Canvas 2D API만 사용하여, `NeonPopSweeper.html` 단일 파일 기반의 완전 플레이 가능한 "Neon Pop Sweeper (Dynamic Vision)" 프로토타입을 구현하세요.

## 기준 문서 (Source of Truth)
- 기획/수치 기준: `docs/01_GAME_SPEC.md`
- 구현 고정 룰: `docs/02_IMPLEMENTATION_RULES.md`
- 핵심 재미 가설: `docs/04_FUN_HYPOTHESES.md`
- 검증 기준(기능 체크리스트/판정 규칙): `docs/05_VALIDATION_CRITERIA.md`

문서 간 충돌이 있을 경우, `01_GAME_SPEC.md`의 수치/공식 값을 우선 적용하세요.

구현 및 검증 단계에서 `04`, `05` 문서 기준을 반드시 점검하세요.

## 하드 제약 조건
1. 출력 파일은 반드시 `NeonPopSweeper.html` 한 개여야 합니다.
2. 외부 라이브러리, 외부 에셋, 외부 폰트, 네트워크 요청을 사용하지 않습니다.
3. 세션 종료 시점은 정확히 `60000ms`여야 합니다. (`performance.now()` 기준)
4. 맵은 `2000x2000`이며 카메라는 맵 바깥을 절대 보여주면 안 됩니다.
5. 플레이어 1명 + AI 봇 3명 구성, 공격/충돌 페널티는 없습니다.
6. 시작 시 시작 패널을 먼저 보여주고 `게임 시작`으로 조인 씬을 연다.
7. 포인터 조이스틱과 키보드(`WASD` + 방향키)를 모두 지원합니다.

## 필수 런타임 동작
- 네온 사각형 닷 스폰/유지/수집
- 별도 성장값(`scaleLevel`) 기반 캐릭터 성장 (콤보와 분리)
- 부드러운 보간 + 클램프를 포함한 동적 카메라 줌
- 중앙 상단 연속 획득 체인 텍스트 (0.6초 체인 윈도우)
- 체인 텍스트의 색상/크기 단계 연출
- 콤보 증가 시 텍스트 스케일 연출은 1회만 재생하고, 최대 배율은 해당 단계 기본 크기의 2배로 적용
- 콤보는 0.6초 내 다음 닷을 획득하지 못하면 즉시 `0`으로 초기화
- 콤보는 연출/점수용으로 유지하고, 캐릭터 스케일은 별도 성장값으로 계산(콤보와 분리)
- 캐릭터 스케일은 최대값 `50`을 넘지 않도록 고정
- 스케일 증가량은 닷 `comboGain`에 축소 비율(`×0.35`)을 적용
- 실시간 랭킹 보드 + 종료 랭킹 패널 + 재시작
- 우측 상단 랭킹 패널은 기존 기본 크기 대비 20% 크게 렌더링
- 랭킹/HUD/결과 패널 점수는 정수(`floor`)로 표시(소수점 금지)
- 좌측 하단 상태 패널(콤보/줌/점수)은 렌더링하지 않음
- 수집 파티클은 캐릭터 중심이 아닌 닷 위치에서 생성(닷 타입별 티어 파라미터, 저사양 시 생성량 완화)
- 현재 빌드에서는 수집 시 화면 흔들림을 사용하지 않음
- 시작 패널: `전체화면` 토글(선택). `게임 시작`은 전체화면을 트리거하지 않음
- 시작 패널: 텍스트(제목/설명/안내)는 중앙 정렬 유지
- 시작 패널: `게임 시작`은 카드 내부 중앙 배치, `전체화면`은 카드 외부 하단 중앙의 별도 버튼으로 배치
- 화면비가 바뀌어도 `전체화면` 버튼은 시작 카드 바로 아래 위치를 유지
- `게임 시작` 후 조인 씬 약 `5초`(진행 바·남은 시간·합류 피드), AI 합류 시각 랜덤, 미합류 봇 비렌더 · 합류 시 화면 밖에서 슬라이드 인
- 조인 씬/게임 시작 전에는 AI 포함 액터가 실제로 이동하지 않음(합류 슬라이드 인만 표시)
- 조인 종료 후 `게임 시작!` HTML 오버레이(`~1.8초`) 뒤 라운드 시작
- 움직이는 보라색 `chaser` 닷 및 붉은 삼각 `hazard` 닷(페널티·분산·발사 구간 비수집) 동작
- 붉은 위험 닷(`hazard`) 동시 활성 개수는 최대 `3`으로 제한
- AI는 위험 닷(`hazard`)을 인지/회피 대상으로 취급하지 않음
- AI 닉네임은 유저형 이름 규칙을 사용하고, 기계형 이름(`Bot_1`, `AI_Player`)은 사용하지 않음
- 액터 닉네임 라벨(YOU + AI)은 월드에서 캐릭터 머리 위에 표시
- 1등 왕관은 닉네임 라벨과 겹치지 않는 높이에 렌더링
- 콤보 텍스트는 `COMBO xN` 형식만 사용하고 불필요한 접미 텍스트(예: `CHAIN n`)를 붙이지 않음

## 구현 순서
1. 캔버스 초기화 + 리사이즈 + 메인 루프 타이밍
2. 상수/상태/유틸 함수 구성
3. 시작 패널/결과 패널 UI 플로우
4. 입력 시스템(포인터 조이스틱 + 키보드)
5. 액터 이동 + AI 타깃/이동 로직
6. 닷 스폰/유지 + 수집 판정
7. 점수/성장 콤보/체인 연출/피버 처리
8. 카메라 줌 + 월드 클램프
9. 렌더 레이어(월드, 닷, 액터, HUD, 체인 텍스트)
10. 재시작 및 성능 저하 대응
11. 검증 로그 출력(완주 여부/오류/재시작) 및 체크리스트 기준 점검

## 명시적 판단 가이드 (임의 해석 금지)
- 아트: 네온 연출보다 플레이 정보 가독성을 우선합니다.
- 밸런스: 체감 이슈가 있어도 `docs/01_GAME_SPEC.md`를 먼저 수정한 뒤 코드 반영합니다.
- UX: 타이머/랭킹/체인 텍스트의 정보 위계를 유지하고, 모바일 터치 오조작 가능성을 우선 제거합니다.
- 성능: 저사양에서 FPS 저하 시 시각효과를 우선 축소하고 핵심 루프 체감은 유지합니다.
- 소셜 체감: 실제 멀티플레이를 암시하는 과장 표현은 피하고, "합류/참가" 중심의 절제된 문구를 사용합니다.
- 렌더 좌표계: 월드 오브젝트(플레이어/닷/파티클)는 동일한 좌표 변환 체인을 사용하고, `renderFrame` 외곽 변환과 오브젝트 변환에서 `viewScale`/`gameView` 오프셋을 중복 적용하지 않습니다.

## 산출물 품질 기준
- 브라우저에서 파일을 열면 즉시 실행 가능해야 함
- TODO/FIXME 플레이스홀더가 없어야 함
- `pointercancel`, `focus/blur` 예외 상황이 안정적으로 처리되어야 함
- Restart를 반복해도 상태가 깨지지 않아야 함
- `게임 시작` 전(시작 패널 표시 상태)에도 undefined 접근 없이 렌더가 안전해야 함
- 초기 레이아웃/리사이즈 단계에서 UI DOM 참조 누락으로 `Cannot read properties of undefined (reading 'style')`가 발생하지 않아야 함
- 시작 전/조인/시작연출/결과 단계에서 랭킹 UI가 노출되지 않아야 함(`playing`에서만 표시)
- UI 스케일은 게임 뷰 비율에 연동되어 축소되며, 최소 스케일 고정으로 인한 과대 표시가 없어야 함
